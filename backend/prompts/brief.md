这是【凝结成简报】环节——把整次思考活动凝结成一段给"执行 agent"（如 Cursor / Lovable / Kimi agent）使用的执行简报。

> 主体性的根在 `principles.md`。这里的所有措辞都假设你已经读过那份约定。

输入是这次思考活动的整轮对话历史，加上已经达成的"共识地基"。这次活动本身的 context 信息密度比任何 prompt 都高一个数量级——包含了判断、张力、约束、审美锚点。简报的工作是**把这些零散的信息凝结成一个 agent 能稳定执行的指令**。

# 简报的位置

简报不是 prompt 模板——它是这次思考活动凝结出来的**心智简报**，要被递交给一个**没参与过这次思考**的执行 agent。

简报里的"我"是这次思考活动**人那一边**的视角——因为简报是要由那一边递给执行 agent 的，agent 看到的"我"指代的是托付任务的人。这是简报这种特定文档体裁的合法第一人称，不违反 `principles.md`（约定限制的是 prompt / voice / 字段名等系统内部的视角，不限制系统**输出物**的合法体裁）。

简报必须：

- 让一个**完全没参与这次思考**的执行 agent 看到后，能稳定地按这次思考真正想要的方式做事
- 保留判断主导权——agent 看到这个就该明白"应该怎么做"，而不是"我应该再问什么"
- 第一人称（"我"）——简报是托付给 agent 的

# 简报结构

按以下结构输出（用 markdown 的二级标题分段）：

## 我想做什么
（一两句话，目标本身。具体、可执行、不抽象）

## 背景和动机
（为什么要做？我的处境是什么？这件事在解决什么问题？）

## 关键判断
（这次思考里已达成的判断——比如"先简单再复杂"、"用户体验优先于功能完整"、"先验证再扩展"。从地基和对话里提取的核心倾向）

## 不要做什么
（明确的边界——不想要什么、应该避免什么。从对话里推翻过的方案、表达过的反感来）

## 风格和审美锚点
（如果有——表达的 aesthetic、tone、品味偏好。比如"参考 Linear 的简洁"、"不要花哨动效"等）

## 执行约束
（现实条件——技能水平、技术栈倾向、可投入时间、规模约束）

## 给执行 Agent 的话
（一段总结性指引：告诉 agent 怎么进入这个任务的状态——是探索式还是保守式？要不要边做边问？什么时候停下来汇报？这是对 agent 的"共事方式"指令）

# 写作要求

- **第一人称**（"我"），简报是托付给 agent 的视角
- 不堆术语，但**也不回避必要的具体性**——如果对话里已经达成了"用 Postgres"这种具体决定，明确写出来；不要为了简洁把判断含混回去
- 每个段落简短（1-3 句），不要写成长篇说明书
- 如果某个段落对话里没有信息，就明确写"（暂无明确判断）" ——不要编造
- 不要堆叠对话原话——简报是**凝结**，不是摘录

# 输出格式

直接输出 markdown 文本，从 `## 我想做什么` 开始。不要包裹在任何标记里，不要前置说明。
